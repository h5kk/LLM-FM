# Feature Memory Backfill

## When to activate

- User says "backfill changelog", "backfill feature memory", "backfill FM"
- User says "generate changelog from git", "update changelog from history"
- User says "populate changelog from commits", "rebuild changelog"
- User types `/backfill`

## Pre-check

Before running, verify that `docs/feature-memory/` exists in the project root:

```bash
ls docs/feature-memory/
```

If it does not exist, stop and tell the user:

> Feature Memory is not initialized in this project. Run "initialize feature memory" first to scaffold the docs structure.

## Step 1: Determine the range

If the user did not specify a range, ask:

> Which commit range should I backfill from?
>
> 1. **Last 48 hours** (default)
> 2. **Last N hours** — e.g. "last 72 hours"
> 3. **Since a date** — e.g. "since 2026-05-01"
> 4. **Since a commit hash** — e.g. "since abc1234"
> 5. **All commits** in the repository
>
> Reply with a number or describe what you want, and I'll run the backfill.

If the user's message already contains a range (e.g. "backfill last 7 days", "backfill --all", "backfill since May 1"), parse it directly and skip the prompt.

**Mapping user input to flags:**

| User says | Flag to use |
|-----------|-------------|
| "last N hours" / "past N hours" | `--hours N` |
| "since YYYY-MM-DD" / "since <date>" | `--since YYYY-MM-DD` |
| "since commit <hash>" / "after <hash>" | `--since-commit <hash>` |
| "all", "--all", "entire history", "everything" | `--all` |
| nothing specified | `--hours 48` (default) |

## Step 2: Run the backfill script

Run from the project root with the chosen flag. The script must be invoked as:

```bash
python plugin/hooks/fm_backfill.py <flag>
```

Examples:
```bash
python plugin/hooks/fm_backfill.py --hours 48
python plugin/hooks/fm_backfill.py --hours 168
python plugin/hooks/fm_backfill.py --since 2026-05-01
python plugin/hooks/fm_backfill.py --since-commit abc1234
python plugin/hooks/fm_backfill.py --all
```

Capture stdout to extract the summary line. The script prints:
- `Processing N commit(s) (...)...`
- `Added N new entries.`
- `Wrote N total entries -> docs/feature-memory/changelogs/changelog.json`
- `Updated docs/feature-memory/changelog-viewer.html` (if viewer was refreshed)
- `Done.`

## Step 3: Report results

Parse the script output and report to the user:

- How many commits were scanned
- How many **new** entries were added (from the "Added N new entries." line)
- Total entries now in changelog.json (from the "Wrote N total entries" line)
- Whether the changelog viewer was updated (look for "Updated" or "Created viewer" in output)
- The commit range that was processed

Example report:

> Backfill complete (last 48 hours):
>
> - Scanned 12 commits
> - Added 7 new changelog entries
> - Changelog now has 23 total entries
> - Viewer updated at `docs/feature-memory/changelog-viewer.html`
>
> Re-run with `--all` to fill older history, or open the viewer to review entries.

If the script prints "No commits found in the specified range.", tell the user there were no commits in that window and suggest widening the range.

If the script exits with an error (e.g. "docs/feature-memory/ not found"), relay the error message and tell the user to run "initialize feature memory" first.

## Notes

- Re-running is safe: the script deduplicates by `(commit_hash, feature_id)` — no double entries.
- Entries generated by this script have `"source": "git-backfill"` and `"review_status": "auto"` in the JSON, which distinguishes them from hook-generated entries.
- The script skips commits that only touch `docs/feature-memory/` itself (meta-updates).
- Unmapped files (not matched by any feature glob in `.feature-memory/config.yaml`) appear as `"feature_id": null` with `"feature_title": "unmapped"`. These are useful for spotting gaps in the feature config.
- **Skip patterns**: generated/compiled/cache files (`*.pyc`, `*.lock`, `node_modules/**`, `dist/**`, etc.) are filtered out automatically. Add project-specific patterns to the `skip_patterns:` list in `.feature-memory/config.yaml`.
- **Dedup**: if the same commit creates multiple entries for the same feature (common when the Stop hook fires), run `/dedup` or `python plugin/hooks/fm_backfill.py --dedup` to merge them.
